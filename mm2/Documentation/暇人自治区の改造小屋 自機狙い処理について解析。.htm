ロックマンの改造や解析なんかを主に書いていきたい。

自機狙い処理について解析。


ここ数日、ロックマン2で使われている自機狙いのための処理について調べていた。
下の方には斜め方向へ動くときに速くなってしまう仕様を修正したパッチもあるよ。

3F175 ... 呼び出したオブジェクトをロックマンの方向に$08~$09の速度にする
2E38C ... ボス用。Xと$2Bで指定したオブジェクトをロックマンの方向に
　　　　　　　　$08~$09の速度にする


三角関数も使ってないのにどうやってロックマンのいる方向に合わせているのだろう？







ロックマン2には割り算のためのルーチンがあった！
個人的に驚き。ずっとないものだと思ってた。

3C871 ... 割り算をする。
　　　　$0B.0A / $0D.0C の結果を$0F.0Eに格納する。

　　　　単純に16bit値の割り算かと思ったら、どうも固定小数点付きの割り算をする
　　　　らしい。だから、割る数に#$00.80とか入れたら2倍とかもできる。

　　　　ただし、入力ミスなのかなんなのか、怪しい処理があって、間違った結果を出す
　　　　こともよくあるようだ。　これは3C89Dを0Bにすると直るらしい。何でだろう。


ここからは3F175からの処理を順に書いていくことにしよう


最初に、オブジェクトとロックマンとの距離を縦、横それぞれ取る。横の距離は
表示上のX座標を使って距離を取ってるので、ページをまたいでいても大丈夫、らしい。

んで、arctanを使って角度を……なんてことはやるはずもなく


縦の距離と横の距離のどっちが長いかで分岐する。
面倒くさいので横の距離が長いときの方だけ書く。

横の距離をLx, 縦の距離をLy, 与えられた速度値を上位下位まとめて
S = (sx, sy)とする。


分岐したら、いきなりオブジェクトの横速度をSにしてしまうようだ。
そこに縦速度を付加したら確実にSより大きな速度になってしまうのだが…

そして、縦速度の設定をする。
要するに縦速度:横速度 = Lx:Lyになればよいのだから、
縦速度に入れる値は、Sを無理やりLy/Lx倍する。

LxをSで割って、その商でLyを割ると、できる。
割り算ルーチンを繰り返すとこんなこともできるんだな―と感動したりした。


あとは、必要に応じて縦速度を反転させたりしておしまい。
縦の距離のほうが長いときは、横と縦を逆にして同じことをやってる。

でも、なんで縦横どっちが遠いかで分岐してるんだろう…?
確かに、分岐しないと結果がおかしくなったけど、なんでだろうなー?
どうも、このへんの話が関係してそうだけど。
・ブレゼンハムのアルゴリズム


と、すっごくテキトーな説明で自機狙いルーチンの解説おわり。

それで、このやり方だとまずいことが1つあって、
斜め方向へ移動することになった場合、移動速度が最大で設定速度の√2倍になる。

当然といえば当然だが、Ly = Lxのとき、(Ly/Lx)・S = 1・S = S
だから移動速度が√(S^2 + S^2) = √(2・S^2) = √(2)・Sになる。


そんな仕様があるから、例えばクイックマンを改造してクイックブーメランを一度に
たくさん出すようにすると…


四角くなる。


それがどうしても気になるのか、後のシリーズではテーブルを使って16方向に狙ったり
するようになったらしい………




今日は長い記事だよーまだあるよー


で、今日はその設定した速度と違う速度になってしまう仕様を、できるだけ修正しようとした
パッチを公開したかったのだった。

どうやって直したかというと……
テーブル作りました。ごめんなさい。

最初に横速度値を設定する時、どうにかして横軸と最終的な方向がなす角をθとしたときの
cosθをSに掛ければ直るはずだから、cosθの値のテーブルを作った。

0 ≦ θ ＜ 45° だから、1 ≦ cosθ ＜ 1/√2
で、割り算ルーチンしかないからcosθの逆数、1～1.41...を用意すればいいんだけど、
整数部は1で固定だから、小数部を少し用意すればいい。

θはどうやって求めるかというと、Ly/Lx = tanθだから、これを計算させて結果の上数bit
をテーブルのインデックスとして使うと、いい感じだった。
（しかし、関数電卓にcos{tan^-1(n/16)} (n = 0, 1, ... , 16)を入力していって16進
に変換する作業は面倒くさかったなー）
